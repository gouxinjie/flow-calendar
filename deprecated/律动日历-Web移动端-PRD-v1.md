# 律动日历 Web 移动端 PRD v1.0

## 1. 文档目的

本版 PRD 只针对 **Web 移动端 H5**，目标是定义当前版本的：

- 页面范围
- 核心功能
- 技术方案

本文档用于收敛范围，避免设计和研发继续按 App / 小程序思路发散。

## 2. 产品定义

- 产品中文名：律动日历
- 产品英文名：Flow Calendar
- 仓库名建议：`flow-calendar-web`
- 包名建议：`flow-calendar`
- 页面标题建议：`Flow Calendar | 律动日历`

### 2.1 一句话

一个面向手机浏览器的、以月历为核心视图的轻量计划管理产品，支持标签排程、模板生成、基础复盘和分享。

### 2.2 本期版本边界

- 终端：手机浏览器优先，兼容移动端 WebView
- 形态：H5 优先
- 不做原生 App
- 不做小程序
- 不依赖系统级能力

## 3. 产品目标

### 3.1 业务目标

- 先验证“月历式轻计划”在移动 Web 上是否成立
- 跑通“创建计划 -> 修改安排 -> 月末复盘 -> 分享”最小闭环
- 在不依赖原生能力的情况下，做出足够顺滑的移动端体验

### 3.2 用户目标

- 用户能在 3 分钟内创建一套月计划
- 用户能在 10 秒内完成某一天的修改
- 用户能在月末生成一张可分享的复盘卡片

## 4. 当前版本功能范围

当前版本只保留 Web 移动端 MVP 必需能力：

- 月历首页
- 日期详情页
- 标签管理
- 模板生成与应用
- 重复规则基础版
- 节假日 / 农历展示
- 情绪 / 体感记录
- 月度统计
- 分享故事卡
- 登录与基础数据同步

当前版本不纳入范围的能力，包括提醒通知、PWA 安装、导入导出、多日历、协同、模板市场、天气联动等。

当前版本采用：

- 登录后使用
- 服务端数据库为主数据源
- 本地仅做缓存，不做游客数据合并
- 账号首次登录成功后默认进入模板页

## 6. 核心功能定义

## 6.1 月历展示

说明：

- 默认按月展示
- 日期格展示公历日期、农历或节气、1 到 2 个主要标签
- 超过 2 个事件显示 `+N`

Web 实现结论：

- 可直接实现

## 6.2 标签事件管理

说明：

- 事件包含标签、标题、备注、状态、时间和可选重复规则
- 支持快捷复制到其他日期
- 事件状态固定为 `pending`、`completed`、`skipped`

时间字段定义：

- 事件支持两种时间类型：`all_day` 与 `scheduled`
- `all_day`：全天事件，不设置具体时间
- `scheduled`：带具体开始时间的事件
- 统计口径中的“已发生事件”仅在 `scheduled` 类型下按具体时间判断
- `all_day` 事件在当天结束前不计入“逾期未完成”

Web 实现结论：

- 可直接实现

## 6.3 模板生成

说明：

- 用户选择主题模板和强度后，系统生成整月计划
- 生成后先预览，再确认应用

模板应用规则：

- 模板默认应用到当前用户唯一日历
- `覆盖`：删除模板作用日期范围内的现有事件后，再写入模板事件
- `合并`：保留原有事件，并将模板事件追加写入相同日期
- 当同一天已有同名同标签事件时，`合并` 不做去重，按两条事件处理
- 模板生成的重复规则按模板原配置写入，不影响日期范围外已有数据

Web 实现结论：

- 可直接实现

## 6.4 重复规则

说明：

- 支持每周几重复、每隔几天重复
- 支持结束日期
- 支持单日例外

交互入口：

- 新增事件时可直接设置重复规则
- 编辑事件时可修改规则
- 修改重复事件时，用户必须二选一：`仅修改这一天` 或 `修改本次及后续`

规则定义：

- `按周重复`：选择周几，如每周一、三、五
- `按间隔重复`：每隔 N 天执行一次
- `开始日期`：规则首次生效日期
- `结束日期`：为空表示持续到用户手动终止
- `单日例外`：仅跳过或改写某一天，不反向修改历史实例

当前不支持：

- 按月重复
- 多套规则叠加
- 复杂 cron 式规则
- 同一事件绑定多条重复规则

Web 实现结论：

- 可直接实现

## 6.5 情绪/体感记录

说明：

- 支持简单标签和短备注
- 用于后续统计和复盘

Web 实现结论：

- 可直接实现

## 6.6 统计与回放

说明：

- 显示分类次数、连续记录、休息比例
- 按时间轴回放本月变化

统计口径：

- 统计周期默认按自然月
- 当查看“当前月”时，仅统计截至当天的已发生事件
- 当查看“历史月”时，统计该月完整自然月
- 分类次数：统计周期内各标签事件总数，不区分完成与否
- 完成率：`completed` 事件数 / 已发生事件总数
- 已发生事件总数：统计周期内满足以下任一条件的事件总数
- 状态为 `completed`
- 状态为 `skipped`
- 事件日期早于统计截止日且状态为 `pending`
- 事件日期等于统计截止日、事件类型为 `scheduled`、开始时间早于当前时间、状态为 `pending`
- 事件日期等于统计截止日、事件类型为 `all_day` 时，在当天结束前不计入分母
- `skipped` 和逾期未完成的 `pending` 事件都计入分母，不计入分子
- 连续记录：连续自然日内，当天至少有 1 个计划事件且该日所有已发生事件均为 `completed`，记为 1 个连续完成日
- 若某天存在计划事件但有任一事件 `skipped` 或逾期未完成，则连续记录中断
- 月历回放：按天展示月内计划和状态变化，不回放事件级操作日志

Web 实现结论：

- 可直接实现

## 6.7 分享故事卡

## 7. 推荐技术方案

本期统一采用：

- `Next.js + Prisma + SQLite`

该组合适合当前 Web 移动端 MVP，原因是：

- 开发速度快
- 部署和运维成本低
- 数据结构清晰，便于后续迁移到 `PostgreSQL`
- 与当前产品复杂度匹配，不必过早引入更重的后端体系

## 7.1 前端

- `Next.js`
- `React`
- `TypeScript`
- UI 以移动端优先
- 样式建议使用 `Tailwind CSS`
- 状态管理建议使用 `Zustand`
- 日期处理建议使用 `dayjs`
- 本地缓存使用 `IndexedDB`

前端职责：

- 月历页、模板页、统计页、我的页等页面渲染
- 日期选择、模板应用、事件编辑等交互
- 月历计算、标签展示、统计图表展示
- 故事卡前端生成

## 7.2 服务端

- 基于 `Next.js` 的 `Route Handlers` 或 `Server Actions`
- ORM 使用 `Prisma`
- 数据库使用 `SQLite`

服务端职责：

- 用户、日历、事件、标签、模板、统计等数据读写
- 模板生成规则处理
- 分享数据生成
- 登录态和基础权限控制

说明：

- `SQLite` 适合当前单应用、低运维成本的 MVP 阶段
- 后续如果出现更强的协同、并发或部署扩展需求，可平滑迁移到 `PostgreSQL`

## 7.3 数据层

- 开发期直接使用本地 `SQLite`
- 生产环境可先采用单实例部署 + `SQLite`
- Prisma 负责 schema 管理与迁移

建议保留清晰的分层：

- 页面层：负责 UI 和交互
- 业务层：负责模板生成、重复规则、统计计算
- 数据层：负责 Prisma 查询与持久化

## 7.4 认证与会话

- 首版采用轻量账号体系
- 建议使用 `Auth.js` 或自定义简单 Session 方案
- 当前版本为登录后使用，不提供游客模式
- 首次访问默认进入登录页
- 若账号无任何日历数据，则登录成功后跳转模板页
- 若账号已有日历数据，则登录成功后默认进入月历页
- 用户数据以服务端 `SQLite` 为主数据源
- `IndexedDB` 仅缓存最近访问的日历数据和模板数据
- 登录前不产生本地业务数据，因此不存在游客数据并入账号的合并流程
- 跨设备同步范围仅包含已登录后写入服务端的数据

## 7.5 H5 关键实现点

- 分享卡采用 Canvas 或 DOM 转图片
- `IndexedDB` 用于本地缓存

## 8. 视觉与色调规范

### 8.1 整体视觉方向

产品色调采用：

- 清透
- 克制
- 有呼吸感
- 轻运动

避免的方向：

- 荧光健身风
- 高饱和运动海报风
- 过度可爱或糖果色日历风

整体气质应更接近：

- 晨间空气感
- 有节奏但不躁
- 轻盈、留白、可长期使用

### 8.2 主色建议

- 主色：`#22C3A6`
- 深主色：`#16967F`
- 浅主色：`#DDF7F1`

主色定位：

- 用于品牌识别
- 用于主按钮、选中态、关键高亮
- 用于表达“律动、健康、流动感”

### 8.3 辅助色建议

- 强调蓝：`#5DA9E9`
- 活力橙：`#FF9F43`
- 柔和紫：`#8B8AEF`
- 灰绿：`#A8B8B0`

建议用途：

- 蓝色：游泳、专注、平静类标签
- 橙色：跑步、训练、高活跃类标签
- 紫色：瑜伽、拉伸、恢复类标签
- 灰绿：休息、低强度、默认中性状态

### 8.4 中性色建议

- 页面背景：`#F7FAF9`
- 卡片背景：`#FFFFFF`
- 主文字：`#1F2A2A`
- 次文字：`#6B7A7A`
- 边框色：`#DCE7E4`

### 8.5 页面应用原则

- 页面背景保持浅色，避免厚重深色底
- 选中日期使用主色描边 + 浅色底，不建议大面积纯色铺满
- 日历事件标签使用多色分类，但整体饱和度应受控
- 休息类标签不使用纯灰，优先使用灰绿
- 统计页和故事卡可适当提升颜色层次，但不能脱离主色体系

### 8.6 视觉关键词

UI 与品牌视觉建议始终围绕以下关键词展开：

- 节奏感
- 呼吸感
- 留白
- 清爽
- 轻运动
- 不焦虑

## 9. UI 设计规范

### 9.1 设计原则

- 优先单手操作，核心点击区尽量落在屏幕中下部
- 信息密度适中，不做满屏堆叠
- 强调层级和留白，不依赖大面积颜色区分结构
- 动效轻量，服务于状态反馈，不做装饰性炫技

### 9.2 栅格与间距

- 页面内容区左右边距建议 `16px`
- 模块之间纵向间距建议 `16px` 到 `24px`
- 卡片内边距建议 `12px` 到 `16px`
- 表单项上下间距建议 `12px`
- 小元素最小点击区不低于 `44px`

### 9.3 排版规范

- 页面主标题：`24px`，半粗
- 区块标题：`18px`，半粗
- 卡片标题：`16px`，中等字重
- 正文：`14px`
- 辅助信息：`12px`
- 数字信息和统计数字可适当提升到 `20px` 到 `28px`

排版要求：

- 优先使用清晰、现代的无衬线字体
- 标题字重高于正文，不依赖纯颜色做层级
- 长段文字控制行宽，避免移动端拥挤

### 9.4 圆角与边框

- 主卡片圆角建议 `20px`
- 次级卡片圆角建议 `16px`
- 标签和小胶囊圆角建议 `10px` 到 `12px`
- 按钮圆角建议 `14px`
- 输入框圆角建议 `14px`
- 默认边框颜色使用 `#DCE7E4`

### 9.5 阴影与层级

- 页面常规卡片使用极轻阴影或无阴影
- 重要浮层使用中等强度阴影
- 不使用厚重投影和多层夸张阴影

建议风格：

- 阴影更像空气感分层，而不是立体拟物

### 9.6 按钮规范

- 主按钮：主色底 + 白字
- 次按钮：白底 + 主色边框 + 主色字
- 文字按钮：无底，强调文字色
- 危险按钮：仅删除类操作使用暖红或低饱和警示色

交互要求：

- 主按钮高度建议 `48px`
- 禁用态降低对比度，不仅仅降低透明度
- 加载态需有明确反馈，避免重复点击

### 9.7 输入与表单规范

- 输入框高度建议 `44px` 到 `48px`
- 默认使用浅边框，不用深描边
- 聚焦态使用主色描边或浅色发光
- 错误态使用低饱和红，不使用刺眼纯红

表单体验要求：

- 同一表单页一次只强调一个主操作
- 复杂配置优先拆为抽屉或分步，而不是堆满一个表单页

### 9.8 标签规范

- 标签优先使用纯色底
- 同一页面标签颜色数建议控制在 4 到 6 种
- 标签文字尽量控制在 2 到 4 个字
- 标签过长时优先截断，不压缩字号
- 标签文字颜色根据背景亮度自动选择深色字或白字，优先保证可读性

建议标签语义：

- 绿色系：默认活跃、常规计划
- 蓝色系：平静、专注、游泳类
- 橙色系：高活跃、跑步、训练类
- 紫色系：恢复、瑜伽、拉伸类
- 灰绿色：休息、暂停、中性状态

### 9.9 日历单元格规范

- 日期格优先保证日期可读性，再展示标签
- 单格内最多展示 `2` 个标签
- 超出使用 `+N` 汇总
- 选中日期使用描边高亮，不建议整格重色填充
- 当天日期需有独立视觉态，但不应压过选中态
- 灰显日期与当月日期要有明确对比

### 9.10 卡片规范

- 首页和统计页均以卡片作为核心容器
- 卡片标题、内容、操作区需有明确层次
- 卡片内不叠过多子卡片，避免层级过深
- 统计卡优先展示 1 个核心数据，不堆太多指标

### 9.11 底部导航规范

- 固定 4 个一级入口
- 图标和文字同时出现
- 当前选中项使用主色高亮
- 底部导航背景保持纯净，不做高饱和渐变

### 9.12 抽屉与弹层规范

- 默认优先使用底部抽屉
- 抽屉圆角建议仅顶部圆角，保持移动端自然手感
- 抽屉头部要有标题和关闭手势提示
- 重要确认操作保留主按钮和取消按钮
- 多步骤配置不建议塞进一个超长抽屉

### 9.13 动效规范

- 页面切换和弹层出现使用短时长过渡
- 推荐时长 `180ms` 到 `240ms`
- 动效曲线以平滑、轻快为主
- 完成、保存、切换状态时要有轻量反馈

避免：

- 大幅弹跳
- 长时间延迟
- 连续旋转和装饰性动效

### 9.14 状态规范

- `pending`：中性或浅色态
- `completed`：主色或高确认态
- `skipped`：低饱和警示态
- `disabled`：弱对比中性色
- `selected`：描边 + 浅底
- `error`：低饱和红
- `success`：主色系成功态

### 9.15 图标规范

- 图标风格保持简洁、圆角、线性优先
- 不混用写实 emoji 风和复杂插画风图标
- 同一页面图标描边粗细保持一致
- 图标只做辅助识别，不替代文字

## 10. 数据结构建议

保留以下核心表即可：

- `users`
- `calendars`
- `event_tags`
- `schedule_events`
- `repeat_rules`
- `daily_checkins`
- `template_plans`

关键字段要求：

- `schedule_events` 必须包含 `status`、`date`、`tag_id`
- `schedule_events` 必须包含 `time_type`
- `schedule_events.start_time` 为可空字段，仅 `scheduled` 类型事件使用
- `schedule_events.repeat_rule_id` 为可空字段，仅重复事件关联规则
- `repeat_rules` 必须包含 `rule_type`、`start_date`、`end_date`
- `repeat_rules.interval_value` 与 `repeat_rules.weekdays` 按 `rule_type` 二选一使用
- `daily_checkins` 必须包含 `mood`、`body_feeling`、`date`

本期可不做：

- 复杂成就表

## 12. 最终结论

如果当前只做 Web 移动端，这个产品依然可以成立，但必须换一个实现策略：

- 把重点放在“计划生成、月历展示、轻量调整、复盘分享”
- 把分享作为闭环的一部分，而不是附属功能
- 避免把版本范围拉向提醒、PWA 或系统级能力

换句话说，Web 版最适合做成一个：

**看起来好用、改起来很快、复盘和分享有吸引力的月历计划工具。**

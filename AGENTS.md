# AGENTS.md

## 1. 文档定位

本文档是 `flow-calendar` 项目的执行约束，面向后续参与该仓库的 Agent、开发者与协作者。

目标只有一个：围绕 PRD 落地 **律动日历（Flow Calendar）Web 移动端 H5 MVP**，避免需求发散、技术跑偏和实现口径不一致。

如无额外说明，后续设计、编码、数据库设计、接口实现、注释、测试与重构都必须遵循本文档。

## 3. 技术方案

### 3.1 固定技术栈

本项目默认技术方案固定为：

- `Next.js`
- `React`
- `TypeScript`
- `Prisma`
- `SQLite`
- `Zustand`
- `dayjs`
- `IndexedDB`

### 3.2 实现原则

- 优先使用 `Next.js App Router`
- 服务端能力优先使用 `Route Handlers` 或 `Server Actions`
- 数据库访问统一通过 `Prisma`
- `SQLite` 为当前主数据源
- `IndexedDB` 仅用于最近访问数据缓存，不是主数据源
- 代码必须可平滑迁移到后续 `PostgreSQL`

## 6. 数据模型约束

### 6.1 核心表

本期核心数据表固定为：

- `users`
- `calendars`
- `event_tags`
- `schedule_events`
- `repeat_rules`
- `daily_checkins`
- `template_plans`

### 6.2 关键字段要求

- `schedule_events` 必须包含：
  - `status`
  - `date`
  - `tag_id`
  - `time_type`
  - `start_time`（可空，仅 `scheduled` 使用）
  - `repeat_rule_id`（可空，仅重复事件使用）
- `repeat_rules` 必须包含：
  - `rule_type`
  - `start_date`
  - `end_date`
  - `interval_value`
  - `weekdays`
- `daily_checkins` 必须包含：
  - `mood`
  - `body_feeling`
  - `date`

### 6.3 数据实现要求

- 禁止编造不存在的表和字段
- 数据库查询必须参数化
- 禁止拼接 SQL
- Prisma schema、迁移和业务类型定义必须同步维护
- 任何枚举值必须与 PRD 一致，不允许临时自创

## 7. 推荐目录结构

如仓库初始化或重构目录结构，优先采用以下组织方式：

```text
src/
├── app/                         # Next.js App Router
│   ├── (main)/                  # 底部导航主路由组
│   ├── login/                   # 登录页
│   ├── account/                 # 账号页
│   ├── tags/                    # 标签管理 / 新建 / 编辑
│   └── api/                     # Route Handlers
├── components/
│   ├── commons/                 # 公共组件
│   └── business/                # 业务组件
├── features/                    # 按业务域拆分
│   ├── calendar/
│   ├── template/
│   ├── stats/
│   ├── tag/
│   └── account/
├── stores/                      # Zustand 状态
├── services/                    # 客户端请求封装
├── server/                      # 服务端业务、鉴权、数据访问
├── lib/                         # 工具、日期、缓存、常量
├── types/                       # TypeScript 类型
└── styles/                      # 全局样式、主题变量、SCSS 资源
prisma/
└── schema.prisma
```

补充要求：

- 公共组件放在 `src/components/commons`
- 复杂业务逻辑优先放在 `src/features` 或 `src/server`
- 统计、模板生成、重复规则等纯逻辑必须避免直接写在页面组件中

## 8. 编码规范

### 8.1 语言与注释

- 所有注释必须使用**中文**
- 用户可见文案默认使用中文，可保留品牌名 `Flow Calendar`
- 组件、函数、复杂逻辑、Prisma 字段含义建议补充中文注释
- 组件头部注释推荐统一格式：

```ts
/**
 * @component 组件名称
 * @description 组件描述
 * @author gouxinjie
 * @created
 * @updated
 */
```

### 8.2 TypeScript 约束

- 必须使用 `TypeScript`
- 禁止使用 `any`
- 所有函数参数、返回值、服务端响应、数据库模型映射必须有明确类型
- 涉及业务规则的枚举、联合类型、DTO 必须单独定义

### 8.3 React / Next.js 约束

- 使用函数组件 + Hooks
- 禁止 class component
- 优先使用 Server Component；涉及浏览器交互时再使用 Client Component
- 页面负责组装，业务逻辑下沉到 `features` / `server` / `lib`
- 列表渲染必须提供稳定 `key`

### 8.4 路径与导入

- 禁止使用 `../../` 这类多层相对路径
- 统一使用 `@/` 路径别名
- 禁止未使用的 import

### 8.5 状态管理

- 局部状态使用 `useState`
- 跨组件共享状态使用 `Zustand`
- 禁止把本应在服务端计算的数据全部塞进全局状态
- 状态命名必须语义化

## 9. 样式与视觉规范

### 9.1 样式策略

本项目样式方案采用：

- 页面结构和常规 UI：优先 `Tailwind CSS`
- 主题变量、复杂样式复用、少量全局样式：使用 `SCSS`

约束如下：

- 禁止污染全局样式
- 禁止大面积内联 `style`
- 同一组件内避免 Tailwind 与 SCSS 无序混用
- 优先沉淀可复用设计 token，而不是硬编码颜色和间距

### 9.2 固定视觉方向

视觉关键词固定为：

- 清透
- 克制
- 呼吸感
- 轻运动
- 留白
- 不焦虑

明确避免：

- 荧光健身风
- 高饱和运动海报风
- 过度可爱或糖果色
- 厚重深色背景

### 9.3 主题色

- 主色：`#22C3A6`
- 深主色：`#16967F`
- 浅主色：`#DDF7F1`
- 强调蓝：`#5DA9E9`
- 活力橙：`#FF9F43`
- 柔和紫：`#8B8AEF`
- 灰绿：`#A8B8B0`
- 页面背景：`#F7FAF9`
- 卡片背景：`#FFFFFF`
- 主文字：`#1F2A2A`
- 次文字：`#6B7A7A`
- 边框色：`#DCE7E4`

### 9.4 移动端 UI 要求

- 以单手操作为优先
- 核心点击区尽量位于屏幕中下部
- 最小点击区不低于 `44px`
- 页面左右边距建议 `16px`
- 模块纵向间距建议 `16px` 到 `24px`
- 卡片圆角建议 `16px` 到 `20px`
- 主按钮高度建议 `48px`
- 抽屉优先使用底部抽屉
- 动效时长建议 `180ms` 到 `240ms`

## 10. API、缓存与服务端约束

### 10.1 API 设计

- 不允许假设接口结构
- 所有接口返回必须有明确类型
- 错误返回统一结构：

```json
{
  "success": false,
  "code": "ERROR_CODE",
  "message": "错误描述（中文）",
  "data": null
}
```

- 成功返回统一结构：

```json
{
  "success": true,
  "code": 200,
  "message": "操作成功",
  "data": {}
}
```

### 10.2 用户身份

- 所有业务请求必须能在服务端上下文中明确解析到 `userId`
- 若通过客户端请求接口，参数、会话或服务端上下文中必须存在并校验 `userId`
- 禁止跳过鉴权直接读写业务数据

### 10.3 缓存策略

- 服务端数据库是主数据源
- `IndexedDB` 仅缓存最近访问的日历和模板数据
- 禁止把缓存当作事实来源覆盖服务端数据
- 登录前不产生本地业务数据

## 11. 安全约束

- 禁止在代码中硬编码 API Key、数据库密码、JWT Secret 等敏感信息
- 所有敏感配置必须使用环境变量
- 所有数据库读写必须参数化
- 所有用户输入必须做校验与必要的 HTML 转义
- 所有 POST 类写操作必须考虑 CSRF 防护方案
- 生产环境禁止输出敏感日志

## 12. 工程质量要求

### 12.1 必须处理的内容

- loading 状态
- error 状态
- 空态
- 删除确认
- 保存反馈
- 异步失败兜底

### 12.2 代码清理

提交前必须移除：

- 未使用变量
- 未使用函数
- 未使用 import
- 未使用样式
- 调试残留

### 12.3 测试建议

以下高风险业务逻辑优先抽成纯函数并补充测试：

- 模板生成
- 重复规则展开与改单逻辑
- 月度统计口径
- 连续记录计算
- 故事卡数据组装

## 13. 开发工作流

### 13.1 新功能

1. 先阅读 PRD 和本 `AGENTS.md`
2. 确认是否在本期范围内
3. 确认接口、数据表和类型定义
4. 先定义类型，再写业务逻辑，再写页面
5. 补齐 loading / error / 空态
6. 自检目录、命名、注释、鉴权与统计口径

### 13.2 修改既有代码

1. 先读原代码
2. 理解当前业务逻辑和数据流
3. 明确影响范围
4. 再进行修改

禁止跳过分析直接改动关键逻辑。

### 13.3 Debug

1. 先定位报错位置
2. 再确认根因
3. 输出修复方案
4. 修复后补充回归验证

## 14. AI 执行约束

后续 Agent 必须遵守：

- 不允许编造接口
- 不允许假设数据库结构
- 不允许擅自增加 PRD 外功能
- 不允许跳过鉴权
- 不允许省略错误处理
- 不允许生成未使用代码
- 不允许修改无关文件
- 不明确的需求必须先询问
- 节假日 / 农历数据必须来自可靠数据源或库，禁止硬编码全年假日表

## 15. 文件操作安全约束

- 禁止批量删除文件或目录
- 禁止使用以下命令：
  - `del /s`
  - `rd /s`
  - `rmdir /s`
  - `Remove-Item -Recurse`
  - `rm -rf`
- 需要删除文件时，只能一次删除一个明确路径的文件
- 如果需要批量删除文件，必须停止并请求用户手动处理
- 仅允许批量删除日志文件，例如 `*.log`、`*.err`

11. 是否没有硬编码敏感信息

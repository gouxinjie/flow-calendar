# 青柠日历 Web 移动端 PRD v2.0

## 1. 文档目的

本版 PRD 只针对 **Web 移动端 H5**，目标是定义当前版本的：

- 页面范围
- 核心功能
- 技术方案

本文档用于把产品收敛为“记录型月历”，不再按健身方案 App 或计划工具展开。

## 2. 产品定义

- 产品中文名：青柠日历
- 产品英文名：Lime Calendar
- 仓库名建议：`lime-calendar-web`
- 包名建议：`lime-calendar`
- 页面标题建议：`Lime Calendar | 青柠日历`

### 2.1 一句话

一个通过月历记录“我做过什么”的轻量生活记录工具，帮助用户随着时间推移回看每天、每周、每月发生过的事情。

### 2.2 核心定位

本产品是：

- 记录工具
- 回顾工具
- 生活轨迹工具

本产品不是：

- 健身方案工具
- 任务管理工具
- 打卡 KPI 工具
- 模板驱动产品

### 2.3 本期版本边界

- 终端：手机浏览器优先，兼容移动端 WebView
- 形态：H5 优先
- 不做原生 App
- 不做小程序
- 不依赖系统级能力

## 3. 产品目标

### 3.1 业务目标

- 验证“月历式生活记录”在移动 Web 上是否成立
- 跑通“记录 -> 回看 -> 搜索 -> 月度回顾”最小闭环
- 做出足够轻量、可持续使用的记录体验

### 3.2 用户目标

- 用户能在 10 秒内完成一条当天记录
- 用户能在 30 秒内找到某个时间段做过的事情
- 用户能通过月历快速知道过去某天或某个月记录过什么

## 4. 当前版本功能范围

当前版本只保留 Web 移动端 MVP 必需能力：

- 月历浏览
- 日期详情页
- 新增 / 编辑 / 删除记录
- 标签管理
- 历史搜索与筛选
- 月度回顾
- 登录与基础数据同步

当前版本不纳入范围的能力，包括模板生成、计划排程、重复规则、提醒通知、导入导出、多日历、协同、模板市场、分享海报等。

当前版本采用：

- 记录后回看，而不是提前规划
- 登录后使用
- 服务端数据库为主数据源
- 本地仅做缓存，不做游客数据合并
- 登录成功后默认进入月历页

## 6. 核心功能定义

## 6.1 月历浏览

说明：

- 默认按月展示
- 日期格展示公历日期、农历或节气、1 到 2 个记录标题摘要
- 超过 2 条记录显示 `+N`
- 支持查看历史月份
- 当前版本仅允许记录当天和过去日期

展示原则：

- 月历格优先回答“这天做过什么”
- 标题摘要优先于标签名称展示
- 若标题过长，优先截断标题，不压缩字号
- 标签仅作为辅助识别信息，不作为主展示内容

Web 实现结论：

- 可直接实现

## 6.2 记录管理

说明：

- 记录包含标题、标签、日期、可选时间、备注
- 支持新增、编辑、删除
- 支持记录到过去日期
- 支持记录到当天
- 支持同一天记录多条内容

时间字段定义：

- 记录支持两种时间类型：`all_day` 与 `scheduled`
- `all_day`：仅记录这一天做过什么，不设置具体时间
- `scheduled`：记录发生在某个具体时间点

记录原则：

- 当前版本只有“已记录”这一种状态
- 不引入 `pending`、`completed`、`skipped` 等计划型状态
- 不要求用户填写完成与否，只记录发生过的事情
- 未来日期不可创建记录

标签规则：

- 标签为可选字段，不作为创建首条记录的前置条件
- 未选择标签的记录按“未分类”处理
- “未分类”为系统展示态，不强制写入独立标签表记录

Web 实现结论：

- 可直接实现

## 6.3 标签管理

说明：

- 用户可创建自定义标签
- 标签支持名称、颜色、图标、排序
- 标签用于提升月历可读性和搜索效率
- 标签为可选，不要求每条记录必须绑定

Web 实现结论：

- 可直接实现

## 6.4 历史搜索与筛选

说明：

- 支持按关键词搜索记录标题和备注
- 支持按标签筛选
- 支持按日期范围筛选
- 默认按日期倒序展示结果
- 支持筛选“未分类”记录

结果规则：

- 搜索结果按天分组展示
- 同一天多条记录按时间排序；无具体时间的记录排在最后
- 空结果时给出空状态，而不是空白页

Web 实现结论：

- 可直接实现

## 6.5 月度回顾

说明：

- 用于帮助用户理解过去一个月记录了什么
- 不做完成率、不做连续打卡、不做任务完成分析

统计口径：

- 统计周期默认按自然月
- 记录总数：该月记录条数
- 记录天数：该月有至少 1 条记录的自然日数量
- 高频标签：该月出现次数最多的标签 Top N
- 最近记录摘要：按时间倒序展示最近若干条记录
- 未分类记录不参与标签排行，但计入记录总数与记录天数

Web 实现结论：

- 可直接实现

## 6.6 登录与基础数据同步

说明：

- 登录后使用
- 数据以服务端为主
- 本地仅做缓存
- 不提供游客模式

跳转规则：

- 首次访问默认进入登录页
- 登录成功后进入月历页

Web 实现结论：

- 可直接实现

## 7. 推荐技术方案

本期统一采用：

- `Next.js + Prisma + SQLite`

该组合适合当前 Web 移动端 MVP，原因是：

- 开发速度快
- 部署和运维成本低
- 数据结构清晰
- 与当前记录型产品复杂度匹配

## 7.1 前端

- `Next.js`
- `React`
- `TypeScript`
- UI 以移动端优先
- 样式建议使用 `Tailwind CSS`
- 状态管理建议使用 `Zustand`
- 日期处理建议使用 `dayjs`
- 本地缓存使用 `IndexedDB`

前端职责：

- 月历页、回顾页、我的页等页面渲染
- 日期选择、记录编辑、标签筛选等交互
- 月历计算、标签展示、统计卡展示

## 7.2 服务端

- 基于 `Next.js` 的 `Route Handlers` 或 `Server Actions`
- ORM 使用 `Prisma`
- 数据库使用 `SQLite`

服务端职责：

- 用户、记录、标签、基础回顾统计等数据读写
- 搜索与筛选查询
- 登录态和基础权限控制

说明：

- `SQLite` 适合当前单应用、低运维成本的 MVP 阶段
- 后续如果出现更强的部署扩展需求，可迁移到 `PostgreSQL`

## 7.3 数据层

- 开发期直接使用本地 `SQLite`
- 生产环境可先采用单实例部署 + `SQLite`
- Prisma 负责 schema 管理与迁移

建议保留清晰的分层：

- 页面层：负责 UI 和交互
- 业务层：负责月历计算、搜索筛选、回顾统计
- 数据层：负责 Prisma 查询与持久化

## 7.4 认证与会话

- 首版采用轻量账号体系
- 建议使用 `Auth.js` 或自定义简单 Session 方案
- 当前版本为登录后使用，不提供游客模式
- 首次访问默认进入登录页
- 登录成功后默认进入月历页
- 用户数据以服务端 `SQLite` 为主数据源
- `IndexedDB` 仅缓存最近访问的月历数据和标签数据
- 登录前不产生本地业务数据，因此不存在游客数据并入账号的合并流程

## 7.5 H5 关键实现点

- `IndexedDB` 用于本地缓存
- 月历网格与记录列表需优先保证移动端流畅度

## 8. 视觉与色调规范

### 8.1 整体视觉方向

产品色调采用：

- 清透
- 克制
- 有呼吸感
- 轻生活感

避免的方向：

- 荧光健身风
- 高饱和海报风
- 过度可爱或糖果色日历风

整体气质应更接近：

- 晨间空气感
- 有节奏但不躁
- 轻盈、留白、可长期使用

### 8.2 主色建议

- 主色：`#5EBF3F`
- 深主色：`#3D9428`
- 浅主色：`#E3F5DA`

主色定位：

- 用于品牌识别
- 用于主按钮、选中态、关键高亮
- 用于表达“流动、轻盈、连续记录感”

### 8.3 辅助色建议

- 强调蓝：`#5DA9E9`
- 活力橙：`#FF9F43`
- 柔和紫：`#8B8AEF`
- 灰绿：`#A8B8B0`

建议用途：

- 蓝色：工作、学习、专注类标签
- 橙色：外出、运动、活动类标签
- 紫色：社交、兴趣、休闲类标签
- 灰绿：休息、居家、默认中性状态

### 8.4 中性色建议

- 页面背景：`#E3F5DA`
- 卡片背景：`#FFFFFF`
- 主文字：`#1F2A2A`
- 次文字：`#6B7A7A`
- 边框色：`#DCE7E4`

### 8.5 页面应用原则

- 页面背景保持浅色，避免厚重深色底
- 选中日期使用主色描边 + 浅色底，不建议大面积纯色铺满
- 日历记录标签使用多色分类，但整体饱和度应受控
- 统计卡与回顾页不做强刺激视觉

### 8.6 视觉关键词

UI 与品牌视觉建议始终围绕以下关键词展开：

- 节奏感
- 呼吸感
- 留白
- 清爽
- 轻生活
- 不焦虑

## 9. UI 设计规范

### 9.1 设计原则

- 优先单手操作，核心点击区尽量落在屏幕中下部
- 信息密度适中，不做满屏堆叠
- 强调层级和留白，不依赖大面积颜色区分结构
- 动效轻量，服务于状态反馈，不做装饰性炫技

### 9.2 栅格与间距

- 页面内容区左右边距建议 `16px`
- 模块之间纵向间距建议 `16px` 到 `24px`
- 卡片内边距建议 `12px` 到 `16px`
- 表单项上下间距建议 `12px`
- 小元素最小点击区不低于 `44px`

### 9.3 排版规范

- 页面主标题：`24px`，半粗
- 区块标题：`18px`，半粗
- 卡片标题：`16px`，中等字重
- 正文：`14px`
- 辅助信息：`12px`
- 统计数字可适当提升到 `20px` 到 `28px`

排版要求：

- 优先使用清晰、现代的无衬线字体
- 标题字重高于正文，不依赖纯颜色做层级
- 长段文字控制行宽，避免移动端拥挤

### 9.4 圆角与边框

- 主卡片圆角建议 `20px`
- 次级卡片圆角建议 `16px`
- 标签和小胶囊圆角建议 `10px` 到 `12px`
- 按钮圆角建议 `14px`
- 输入框圆角建议 `14px`
- 默认边框颜色使用 `#DCE7E4`

### 9.5 阴影与层级

- 页面常规卡片使用极轻阴影或无阴影
- 重要浮层使用中等强度阴影
- 不使用厚重投影和多层夸张阴影

### 9.6 按钮规范

- 主按钮：主色底 + 白字
- 次按钮：白底 + 主色边框 + 主色字
- 文字按钮：无底，强调文字色
- 危险按钮：仅删除类操作使用暖红或低饱和警示色

交互要求：

- 主按钮高度建议 `48px`
- 禁用态降低对比度，不仅仅降低透明度
- 加载态需有明确反馈，避免重复点击

### 9.7 输入与表单规范

- 输入框高度建议 `44px` 到 `48px`
- 默认使用浅边框，不用深描边
- 聚焦态使用主色描边或浅色发光
- 错误态使用低饱和红，不使用刺眼纯红

表单体验要求：

- 同一表单页一次只强调一个主操作
- 复杂配置优先拆为抽屉，而不是堆满一个页面

### 9.8 标签规范

- 标签优先使用纯色底
- 同一页面标签颜色数建议控制在 4 到 6 种
- 标签文字尽量控制在 2 到 4 个字
- 标签过长时优先截断，不压缩字号
- 标签文字颜色根据背景亮度自动选择深色字或白字，优先保证可读性

记录摘要规范：

- 月历格中的记录摘要优先展示标题前 1 行
- 标题摘要建议控制在 6 到 8 个汉字内的视觉长度
- 同一天多条记录时，按创建时间倒序展示前 2 条
- 若存在具体时间，则优先展示更晚创建或更晚发生的记录

建议标签语义：

- 绿色系：常规生活记录
- 蓝色系：工作、学习、专注
- 橙色系：外出、活动、运动
- 紫色系：社交、兴趣、休闲
- 灰绿色：休息、居家、中性状态

### 9.9 日历单元格规范

- 日期格优先保证日期可读性，再展示标签
- 单格内最多展示 `2` 个标签
- 超出使用 `+N` 汇总
- 选中日期使用描边高亮，不建议整格重色填充
- 当天日期需有独立视觉态，但不应压过选中态
- 灰显日期与当月日期要有明确对比

### 9.10 卡片规范

- 月历页和回顾页均以卡片作为核心容器
- 卡片标题、内容、操作区需有明确层次
- 卡片内不叠过多子卡片，避免层级过深
- 回顾卡优先展示 1 个核心信息，不堆太多指标

### 9.11 底部导航规范

- 固定 3 个一级入口
- 图标和文字同时出现
- 当前选中项使用主色高亮
- 底部导航背景保持纯净，不做高饱和渐变

### 9.12 抽屉与弹层规范

- 默认优先使用底部抽屉
- 抽屉圆角建议仅顶部圆角，保持移动端自然手感
- 抽屉头部要有标题和关闭手势提示
- 重要确认操作保留主按钮和取消按钮

### 9.13 动效规范

- 页面切换和弹层出现使用短时长过渡
- 推荐时长 `180ms` 到 `240ms`
- 动效曲线以平滑、轻快为主
- 保存、切换月份、应用筛选时要有轻量反馈

避免：

- 大幅弹跳
- 长时间延迟
- 连续旋转和装饰性动效

### 9.14 状态规范

- `default`：默认中性色
- `selected`：描边 + 浅底
- `empty`：弱对比中性色
- `error`：低饱和红
- `success`：主色系成功态

### 9.15 图标规范

- 图标风格保持简洁、圆角、线性优先
- 不混用写实 emoji 风和复杂插画风图标
- 同一页面图标描边粗细保持一致
- 图标只做辅助识别，不替代文字

## 10. 数据结构建议

保留以下核心表即可：

- `users`
- `activity_tags`
- `activity_logs`
- `user_settings`

关键字段要求：

- `activity_logs` 必须包含 `user_id`、`date`、`title`
- `activity_logs` 必须包含 `time_type`
- `activity_logs.start_time` 为可空字段，仅 `scheduled` 类型记录使用
- `activity_logs.tag_id` 为可空字段
- `activity_logs.note` 为可空字段
- `activity_tags` 必须包含 `name`、`color`、`sort_order`
- `user_settings` 至少包含 `user_id` 与基础显示设置

本期可不做：

- 复杂成就表
- 模板表
- 重复规则表

## 12. 最终结论

如果当前只做 Web 移动端，`青柠日历` 最适合做成一个：

- 以月历为核心的记录工具
- 用来知道“之前做过什么”的回顾工具
- 而不是健身计划、任务打卡或模板产品

换句话说，它最重要的价值不是“安排未来”，而是：

**把已经发生过的生活，清晰地留在月历上。**
